## 一、多团队共性设计模式提炼

### 1.1 三团队对比分析

| 维度 | 学术论文（39 Agent） | 专利申请（42 Agent） | 漫剧生成（34 Agent） |
|------|---------------------|---------------------|---------------------|
| **管控层** | PM(导师)+Orchestrator | PM(专利总监)+Orchestrator | PM(主编)+Orchestrator |
| **Phase 数** | 7（含研究定位） | 7（含挖掘+模拟） | 6（含视觉设定+排版） |
| **Gate 数** | 7 | 7 | 6 |
| **贯穿 Agent** | 5（导师+诚信+一致+翻译+记忆） | 5（法规+时限+一致+对标+知识） | 4（风格+叙事+角色+记忆） |
| **核心挑战** | 学术严谨性 | 法规合规+模拟对抗 | 角色一致性+画风统一 |
| **交付物** | 论文+投稿材料 | 专利文件+审查答复 | 漫剧成品+多平台格式 |

### 1.2 五大共性设计模式

```
┌───────────────────────────────────────────────────────────────────┐
│  🧬 多 Agent 团队五大共性设计模式                                    │
├───────────────────────────────────────────────────────────────────┤
│                                                                    │
│  模式1: 管控分离（Separation of Concerns）                          │
│  ├─ PM = 战略决策者（方向审定+里程碑审查+终审验收）                    │
│  └─ Orchestrator = 执行协调者（任务分解+Agent调度+冲突仲裁）           │
│                                                                    │
│  模式2: 阶段门控（Phase-Gate Pipeline）                             │
│  ├─ 每个 Phase 有明确的输入/输出定义                                │
│  ├─ Phase 间必须通过质量门控（Gate）才能推进                         │
│  └─ Gate 不通过 → 回退到前序阶段修正                                 │
│                                                                    │
│  模式3: 贯穿守护（Cross-Phase Guardians）                          │
│  ├─ 全程存在的"守护者"Agent，持续监控特定维度                       │
│  ├─ 贯穿 Agent 数量通常 ≤ 5 个（经验法则）                          │
│  └─ 贯穿 Agent 不做执行，只做检查/监督/记忆                          │
│                                                                    │
│  模式4: 渐进交付（Progressive Delivery）                            │
│  ├─ 每个 Phase 都产出可检查的中间成果                               │
│  ├─ 后续 Phase 可基于前序产出独立迭代                                │
│  └─ 支持"回溯修正"而非"从头开始"                                    │
│                                                                    │
│  模式5: 场景适配（Scenario Mapping）                                │
│  ├─ 通用骨架 + 专用 Agent 池                                       │
│  ├─ 通过"适配矩阵"选择启用哪些 Agent                                │
│  └─ 不同场景可复用骨架和通信协议                                     │
│                                                                    │
└───────────────────────────────────────────────────────────────────┘
```

### 1.3 贯穿型 Agent 派生规则

| 基础维度 | 学术论文 | 专利申请 | 漫剧生成 | **通用模板** |
|---------|---------|---------|---------|------------|
| 一致性守护 | 术语/符号统一 | 术语/标记统一 | 画风/角色统一 | **产出物一致性** |
| 记忆管理 | 研究决策记录 | 审查经验沉淀 | 创作决策记录 | **过程决策记忆** |
| 专业监督 | 学术诚信监督 | 法规合规监督 | — | **领域合规监督** |
| 反馈模拟 | 导师质疑模拟 | 审查员模拟 | — | **外部评审模拟** |
| 语言支持 | 双语翻译 | — | — | **语言规范守护** |

> **核心规律**：任何领域的 Agent 团队都至少需要 **一致性守护 + 过程记忆** 两个贯穿 Agent。

## 二、可复用论文写作团队骨架

### 2.1 骨架定义

```
╔═══════════════════════════════════════════════════════════════╗
║                    🎯 PM Agent（{领域}主管）                    ║
║          方向审定 │ 里程碑审查 │ 质量终审 │ 最终验收            ║
╚═════════════════════════╤═══════════════════════════════════╝
                            │
╔═══════════════════════════▼═════════════════════════════════╗
║                🔄 Orchestrator（{领域}协调员）                 ║
║         任务分解 │ Agent调度 │ 进度管控 │ 冲突仲裁 │ 回退管理   ║
╚═══════════════════════════╤═══════════════════════════════════╝
                            │
  ┌──────────┬───────────┬──────────┬──────────┬──────────┐
  ▼          ▼           ▼          ▼          ▼          ▼
 Phase 0   Phase 1    Phase 2    Phase 3    Phase 4    Phase 5
 {定位}   {基础}      {核心}      {深化}      {交付}      {审校}
  [G0]     [G1]        [G2]        [G3]        [G4]        [G5]

  ═══════════ 贯穿型 Agent（固定 ≥ 2 个） ═══════════
  ┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐
  │ C1 一致性守护者    │ │ C2 过程记忆管理    │ │ C3 {领域}监督者  │
  └──────────────────┘ └──────────────────┘ └──────────────────┘
```

### 2.2 通用 Phase 模板

| Phase | 通用职责 | 学术论文映射 | 专利申请映射 | 漫剧生成映射 |
|-------|---------|-------------|-------------|-------------|
| **Phase 0** | 定位/策划 | 研究定位（Scope→Gap→Hypo） | 专利挖掘（创新捕获→检索） | 主题策划（主题→世界观→受众） |
| **Phase 1** | 基础建设 | 文献工程（检索→评估→综合） | 布局策略（组合→类型→地域） | 剧本创作（故事→角色→分镜） |
| **Phase 2** | 核心执行 | 方法论设计+研究执行 | 专利文件撰写 | 视觉设定 |
| **Phase 3** | 深化完善 | 论证构建+论文写作 | 内部质量审核 | 画面生成 |
| **Phase 4** | 精化交付 | 审校与投稿准备 | 模拟审查+提交 | 排版合成 |
| **Phase 5** | 最终审校 | 格式+投稿材料 | 批量管理+运营 | 审校+发布 |

### 2.3 调度策略通用规则

```yaml
# Phase 内并行规则
phase_parallelism:
  - 同一 Phase 内无数据依赖的 Agent 可并行执行
  - 有依赖关系的 Agent 必须串行
  - 并行 Agent 数量建议 ≤ 3（避免通信复杂度爆炸）
  
# Gate 回退规则
gate_rollback:
  minor_issue: "当前 Phase 内修复"     # 小问题：当前 Phase 修复
  major_issue: "回退到前序 Phase"       # 大问题：回退到依赖链起点
  critical_issue: "升级至 PM 决策"      # 关键问题：PM 介入决策
  
# Agent 活跃度规则
agent_lifecycle:
  phase_agent: "仅在所属 Phase 激活"
  cross_agent: "全程激活但分阶段强化"
  gatekeeper: "仅在 Gate 检查时激活"
```

## 三、如何基于骨架派生专用团队

### 3.1 派生四步法

```
Step 1: 定义领域 → 明确 PM 角色定位和核心挑战
Step 2: 映射 Phase → 将通用 Phase 模板映射到领域具体任务
Step 3: 设计 Agent → 每个 Phase 拆分为 3-6 个专业化 Agent
Step 4: 配置 Gate → 为每个 Gate 设计领域特定的检查清单
```

### 3.2 Agent 拆分原则

| 原则 | 说明 | 反例（避免） |
|------|------|-------------|
| **单一职责** | 每个 Agent 只做一件事 | "分析+写作+审核"合为1个 Agent |
| **输入输出明确** | 每个 Agent 有清晰的输入/输出定义 | 模糊的"处理数据"职责 |
| **可检查性** | 每个 Agent 的产出可被 GateKeeper 检查 | 无法量化的"改善"产出 |
| **可替换性** | 单个 Agent 可被同层其他 Agent 替代 | 唯一不可替代的"神级" Agent |
| **粒度适中** | Phase 内 Agent 数量 3-6 个 | 1个（太粗）或 >8 个（太细） |

### 3.3 贯穿 Agent 配置矩阵

| 场景类型 | C1 一致性 | C2 记忆 | C3 领域监督 | C4 反馈模拟 | C5 语言 |
|---------|----------|--------|------------|------------|--------|
| 学术论文 | ✅术语符号 | ✅研究决策 | ✅学术诚信 | ✅导师模拟 | ✅双语翻译 |
| 专利申请 | ✅术语标记 | ✅审查经验 | ✅法规追踪 | ✅审查员模拟 | ❌ |
| 漫剧生成 | ✅画风角色 | ✅创作决策 | ❌ | ❌ | ❌ |
| 咨询报告 | ✅术语数据 | ✅研究决策 | ❌ | ✅客户模拟 | ❌ |
| 技术选型 | ✅术语指标 | ✅决策记录 | ❌ | ✅利益方模拟 | ❌ |

## 四、核心可复用组件库

### 4.1 PM Agent 通用 Prompt 模板

```markdown
## 角色：资深{领域}专家/主管
## 核心职责
1. 审定{领域}方向和目标
2. 审查每个 Phase 的里程碑产出
3. 处理 Gate 未通过的升级决策
4. 最终验收交付物

## 审查维度
- {领域}方向是否准确
- 方法/策略是否合理
- 产出质量是否达标
- 是否有遗漏的重大风险

## 行为模式
- 苏格拉底式提问 + 批判性审查
- 发现问题时给出方向性指引（而非具体操作）
```

### 4.2 Orchestrator 通用 Prompt 模板

```markdown
## 角色：{领域}项目协调员
## 当前任务
- 课题：{课题描述}
- 当前阶段：Phase {N}
- 已完成成果：{前序阶段摘要}

## 调度策略
1. 分析当前 Phase 需要哪些 Agent 参与
2. 确定执行顺序和并行关系
3. 定义每个 Agent 的输入/输出
4. 设置 Gate 检查标准

## 冲突处理
当多个 Agent 结论矛盾时：
1. 列出各方观点和证据
2. 评估证据强度
3. 做出裁决并说明理由
```

### 4.3 GateKeeper 通用检查框架

```markdown
## Gate {N} 检查清单

### 完整性
- [ ] 所有预期产出物是否齐全
- [ ] 数据/信息是否覆盖预设范围

### 一致性
- [ ] 与前序 Phase 结论是否矛盾
- [ ] 术语/格式/数据口径是否一致

### 质量
- [ ] 核心结论是否有充分证据支撑
- [ ] 是否存在明显的逻辑漏洞
- [ ] 质量评分是否达到阈值（≥70/100）

### 风险
- [ ] 是否识别了主要风险
- [ ] 风险是否有应对方案
```

## 五、通用 Gate 模板

### 5.1 Gate 评分体系

```yaml
gate_rating:
  excellent:  # 90-100
    action: "直接通过，记录最佳实践"
  good:       # 80-89
    action: "通过，标注改进建议"
  acceptable: # 70-79
    action: "有条件通过，需在下一 Phase 前修补"
  poor:       # <70
    action: "不通过，回退修正"

gate_pass_threshold: 70  # 最低通过分数
gate_record: true        # 是否记录 Gate 结果
```

### 5.2 Gate 决策树

```
Gate 检查
    │
    ├─ 评分 ≥ 90 → ✅ 通过（优秀）
    │
    ├─ 70 ≤ 评分 < 90 → ⚠️ 有条件通过
    │   ├─ 影响范围小 → 标注，继续推进
    │   └─ 影响范围大 → 修补后再推进
    │
    └─ 评分 < 70 → ❌ 不通过
        ├─ 问题在当前 Phase → 当前 Phase 内修复
        ├─ 问题来自前序 Phase → 回退到前序 Phase
        └─ 问题需要战略决策 → 升级至 PM
```

## 六、通用消息协议

### 6.1 标准 Agent 间消息格式

```json
{
  "message_id": "MSG-{PROJECT}-{TIMESTAMP}-{SEQ}",
  "type": "phase_output | query | feedback | gate_result | rollback",
  "from": { "agent_id": "A{N}_{Name}", "phase": "Phase{M}" },
  "to": { "agent_id": "A{X}_{Name}|Orchestrator|PM", "phase": "Phase{Y}|null" },
  "payload": {
    "summary": "一句话摘要",
    "key_findings": ["发现1", "发现2"],
    "artifacts": ["产出物1.md", "产出物2.xlsx"],
    "quality_flag": "green | yellow | red"
  },
  "gate_status": "pass | conditional_pass | fail",
  "rollback": {
    "required": false,
    "target_phase": null,
    "reason": null
  }
}
```

### 6.2 Phase 产出物标准目录结构

```
{project_id}/
```

## 八、已派生团队对照

| 维度 | 通用骨架 | 学术论文 | 专利申请 | 漫剧生成 |
|------|---------|---------|---------|---------|
| Phase 数 | 6 | **7**（+研究定位） | **7**（+挖掘+模拟） | **6** |
| Agent 总数 | 骨架12-20 | **39** | **42** | **34** |
| Gate 数 | 6 | **7** | **7** | **6** |
| 贯穿 Agent | 2-3 | **5** | **5** | **4** |
| Phase 内最大并行 | 3 | 3 | 5 | 4 |
| 回退深度 | 1-2 Phase | 2 Phase | 2 Phase | 1 Phase |

> **经验总结**：骨架提供通用结构，专用团队通过 Phase 映射 + Agent 拆分 + Gate 定制来适配具体领域。Agent 总量与领域复杂度正相关，但贯穿 Agent 始终 ≤ 5 个。
