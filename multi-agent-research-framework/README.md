## 一、架构总览与设计理念

### 1.1 原始架构回顾

原始架构（晓辉博士版本）的核心贡献：
- ✅ **分层分阶段协作**：PM → Orchestrator → Phase 1~4 → Review
- ✅ **贯穿型智能体**：TheoryAgent（第一性原理）+ ExplorerAgent（前沿追踪）
- ✅ **专业化分工**：不同 Agent 聚焦不同维度

### 1.2 优化升级点

| 维度 | 原始架构 | 优化后架构 |
|------|---------|-----------|
| **信息来源** | 直接从分析开始 | 新增 **Phase 0: 信息采集层** |
| **交付能力** | 合成层结束 | 新增 **Phase 5: 专业交付层** |
| **质量控制** | 仅最终 Review | **每阶段 GateKeeper 门控** |
| **过程记忆** | 无 | 新增 **MemoryAgent** |
| **反馈机制** | 单向流程 | **双向反馈循环** |
| **事实校验** | 无 | 新增 **FactCheckAgent** |
| **工具使用** | 未明确 | 新增 **Tool Pool 基础设施** |
| **场景适配** | 固定角色 | **通用骨架 + 场景映射** |

### 1.3 核心设计原则

```
┌─────────────────────────────────────────────────┐
│  7 大设计原则                                     │
├─────────────────────────────────────────────────┤
│  1. 分层管控  — PM > Orchestrator > GateKeeper   │
│  2. 阶段门控  — 每Phase结束必须通过质量检查         │
│  3. 反馈循环  — 验证失败可回溯至任意前序阶段         │
│  4. 记忆管理  — 全程维护上下文与中间成果             │
│  5. 并行执行  — 同Phase内Agent可并发工作            │
│  6. 渐进交付  — 支持Phase级中间成果输出             │
│  7. 弹性扩展  — Agent角色池按需增减                 │
└─────────────────────────────────────────────────┘
```

### 2.2 各 Agent 详细定义

#### Layer 0: 管控层

| Agent | 角色 | 核心职责 | 触发条件 |
|-------|------|---------|---------|
| **PM Agent** | 项目经理 | 1. 确认研究方向与目标<br>2. 审查里程碑成果<br>3. 最终验收交付物 | 项目启动/每阶段结束/项目完成 |
| **Orchestrator** | 协调器 | 1. 将目标分解为子任务<br>2. 调度 Agent 执行顺序<br>3. 裁决探索方向分歧<br>4. 资源分配与冲突仲裁 | 全程持续运行 |

#### Layer 1: 质量门控

| Agent | 角色 | 核心职责 |
|-------|------|---------|
| **GateKeeper** | 门控员 | 每Phase结束时检查：<br>1. 产出是否满足该阶段目标<br>2. 数据质量是否达标<br>3. 逻辑是否自洽<br>**不通过 → 回退修正** |

#### Phase 1: 信息采集层（Collection）

| Agent | 角色 | 核心能力 | 工具依赖 |
|-------|------|---------|---------|
| **A1 SearchAgent** | 广域搜索者 | 网络搜索、数据库查询、API调用 | Web Search, Database Query |
| **A2 DataAgent** | 数据采集员 | 结构化数据获取、清洗、标准化 | Data Parser, ETL Tools |
| **A3 DomainAgent** | 领域专家 | 领域知识注入、专业术语解释、背景补充 | Knowledge Base |

#### Phase 2: 深度分析层（Analysis）

| Agent | 角色 | 核心能力 | 分析方法 |
|-------|------|---------|---------|
| **A4 StructureAgent** | 结构化分析师 | 框架构建、分类梳理、层次解构 | MECE、分类树、流程图 |
| **A5 CompareAgent** | 对比分析师 | 多维度对比、差异识别、优劣评估 | 矩阵对比、雷达图、SWOT |
| **A6 QuantAgent** | 量化分析师 | 数据建模、指标计算、统计推断 | 回归分析、聚类、时序分析 |
| **A7 CausalAgent** | 因果推理师 | 因果关系识别、影响路径分析 | 因果图、系统动力学、归因分析 |

#### Phase 3: 综合洞察层（Synthesis）

| Agent | 角色 | 核心能力 |
|-------|------|---------|
| **A8 InsightAgent** | 洞察提炼师 | 模式识别、趋势预测、规律发现 |
| **A9 StrategyAgent** | 策略设计师 | 方案设计、路径规划、建议输出 |

#### Phase 4: 验证层（Validation）

| Agent | 角色 | 核心能力 | 检查维度 |
|-------|------|---------|---------|
| **A10 CriticAgent** | 批判性审查员 | 逻辑漏洞检测、偏见识别、反面论证 | 逻辑性、客观性、完整性 |
| **A11 FactCheckAgent** | 事实核查员 | 数据源验证、交叉检证、引用溯源 | 准确性、可靠性、时效性 |

#### Phase 5: 交付层（Delivery）

| Agent | 角色 | 核心能力 | 输出格式 |
|-------|------|---------|---------|
| **A12 WriterAgent** | 报告撰写师 | 结构化写作、多风格适配、逻辑叙事 | MD/DOCX/PDF/HTML |
| **A13 VisualAgent** | 可视化设计师 | 图表设计、信息图生成、演示文稿 | Chart/Infographic/PPT |

#### 贯穿型智能体（Cross-Phase Agents）

| Agent | 角色 | 核心能力 | 活跃阶段 |
|-------|------|---------|---------|

### 2.3 阶段间信息流

```
Phase 1 ─────────────────────────────────────────────▶ Phase 2
  输出：原始数据集、文献库、领域知识图谱                    输入：↑
  质量要求：覆盖率≥80%，来源≥3个独立渠道

Phase 2 ─────────────────────────────────────────────▶ Phase 3
  输出：分析报告集（结构/对比/量化/因果）                   输入：↑
  质量要求：每个维度有数据支撑，MECE无遗漏

Phase 3 ─────────────────────────────────────────────▶ Phase 4
  输出：核心洞察列表、策略建议方案                         输入：↑
  质量要求：洞察有分析支撑，策略可执行

Phase 4 ─────────────────────────────────────────────▶ Phase 5
  输出：验证报告（逻辑审查+事实核查）                      输入：↑
  质量要求：所有事实可溯源，逻辑链完整

Phase 5 ─────────────────────────────────────────────▶ 最终交付
  输出：完整研究报告 + 可视化材料
  质量要求：格式规范、引用完整、可读性强
```

### 2.4 反馈循环机制

```
                    ┌──── 回退路径 ────┐
                    │                  │
Phase 1 → Phase 2 → Phase 3 → Phase 4 ─┤→ Phase 5
  ▲          ▲          ▲              │
  │          │          │              │
  └──────────┴──────────┴──── 问题回溯 ─┘

触发条件：
• 事实核查发现数据错误 → 回退至 Phase 1 重新采集
• 逻辑审查发现分析偏差 → 回退至 Phase 2 补充分析
• 洞察不够深入 → 回退至 Phase 3 重新综合
```

### 3.3 咨询报告质量标准

```
┌─────────────────────────────────────────────────────────┐
│  📋 McKinsey式报告质量检查清单                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ☐ MECE：论点相互独立、完全穷尽                            │
│  ☐ 金字塔原理：结论先行、归类分组、逻辑递进                  │
│  ☐ So What?：每个分析都有明确启示                          │
```

### 10.6 人像克隆（角色一致性）方案

| 方案 | 费用 | 原理 | 效果 | 推荐度 |
|------|------|------|------|-------|
| **SD + InstantID** | **¥0** | InsightFace 人脸embedding + IP-Adapter + ControlNet | 单图即可高保真复制人脸，多风格多姿态 | ⭐⭐⭐⭐⭐ **默认推荐** |
| **SD + PuLID** | **¥0** | Pure and Lightning ID Customization | 更精准的 ID 保持，风格迁移更自然 | ⭐⭐⭐⭐⭐ |
| **SD + IP-Adapter-FaceID Plus v2** | **¥0** | 从参考图提取面部特征 → 转移到不同姿态 | 灵活度高，多视角一致 | ⭐⭐⭐⭐ |
| 混元生图 + LoRA 训练 | 付费（训练+推理） | 针对特定角色训练 LoRA | 一致性极高但成本高 | ⭐⭐⭐ |

```
🎨 人像克隆默认推荐：SD + InstantID（开源免费 + 单图即可克隆）
   需要：SDXL 大模型 + InstantID 插件 + InsightFace
   流程：
   1. 提供一张角色参考图（正脸清晰）
   2. InstantID 提取人脸 embedding + 面部标志
   3. 结合 IP-Adapter + ControlNet 生成任意姿态/场景
   4. 角色外观在不同分镜中保持高度一致
```
